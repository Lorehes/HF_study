# 🛒 고객 문의 유형 분류기

K쇼핑 콜센터 질의응답 데이터를 활용하여, 고객 문의 텍스트를 **6개 유형**(AS, 결제, 교환, 반품, 배송, 주문)으로 자동 분류하는 AI 분류기입니다.

## 프로젝트 구조

```
고객문의_분류기/
├── 고객_문의_유형_분류기.ipynb   # 메인 노트북 (전처리 → 학습 → 예측)
├── .env                         # 데이터 경로 설정 (git 제외)
└── README.md
```

## 사용 데이터

- **출처:** AI Hub 민원(콜센터) 질의응답 데이터셋 — K쇼핑 도메인
- **형식:** ZIP → JSON (대화셋 기반 콜센터 통화 녹취 텍스트)
- **카테고리:** AS, 결제, 교환, 반품, 배송, 주문 (6개)

이 프로젝트는 3단계에 걸쳐 모델을 개선했습니다. 각 단계의 변경 사항, 근거, 결과를 상세히 기록합니다.

### v1 → v2: 데이터 보강 & 분류기 비교 (+3.6%p)

#### 변경 내용

| 항목          | v1                      | v2                                       |
| ------------- | ----------------------- | ---------------------------------------- |
| 입력 텍스트   | 고객 발화만             | 고객 발화 + 상담원 응답 + 고객 의도 결합 |
| 학습 데이터   | 500건/카테고리          | 1,000건/카테고리                         |
| 테스트 데이터 | 100건/카테고리          | 200건/카테고리                           |
| 분류기        | LogisticRegression 단독 | LogisticRegression vs SVC(RBF) 비교      |
| 평가          | 분류 리포트만           | 분류 리포트 + 혼동 행렬 시각화           |

#### 개선 근거

1. **입력 텍스트 보강**: 원본 데이터의 JSON에는 `고객질문(요청)` 외에도 `상담사답변`, `고객의도` 필드가 존재합니다. 고객 발화만으로는 "확인 좀 해주세요"처럼 맥락이 부족한 문의가 많아 분류가 어렵습니다. 상담원 응답(최대 3개)과 고객 의도 라벨을 `[상담]`, `[의도]` 태그로 결합하여 맥락을 풍부하게 했습니다.

2. **학습 데이터 증가**: 카테고리별 500건은 7개 카테고리를 구분하기에 부족합니다. 1,000건으로 늘려 분류기가 더 다양한 표현 패턴을 학습할 수 있게 했습니다.

3. **SVM 비교 추가**: 384차원 임베딩 벡터는 고차원 공간이므로, SVM(RBF 커널)이 비선형 결정 경계를 학습하여 더 나은 성능을 보일 가능성이 있었습니다.

#### v2 결과 분석

- 전체 정확도: 63.6% (+3.6%p)
- SVM(62.9%)보다 LogisticRegression(63.6%)이 더 높음 → 현재 데이터에서는 선형 경계가 더 적합
- **주문(37.0%)과 업무처리(43.0%)가 여전히 미흡** — 혼동 행렬에서 업무처리→결제(33건), 업무처리→교환(29건), 주문→결제(42건), 주문→배송(39건)으로 대량 오분류 발생

#### 이 방법을 선택한 이유

기존 파이프라인(사전학습 임베딩 + 전통 분류기) 구조를 유지하면서 가장 빠르게 시도할 수 있는 개선이었습니다. 모델 교체나 파이프라인 변경 없이 **데이터 품질**과 **분류기 선택**만으로 얼마나 성능이 오르는지 확인하는 것이 목적이었습니다.

### v2 → v3: 임베딩 모델 업그레이드 & 카테고리 재정의 (+10.3%p)

#### 변경 내용

| 항목          | v2                                                | v3                                                |
| ------------- | ------------------------------------------------- | ------------------------------------------------- |
| 임베딩 모델   | `paraphrase-multilingual-MiniLM-L12-v2` (384차원) | `paraphrase-multilingual-mpnet-base-v2` (768차원) |
| 카테고리      | 7개 (AS, 결제, 교환, 반품, 배송, 업무처리, 주문)  | 6개 (AS, 결제, 교환, 반품, 배송, 주문)            |
| 제거 카테고리 | -                                                 | **업무처리**                                      |

#### 개선 1: 임베딩 모델 교체

**문제**: MiniLM-L12은 384차원 벡터를 생성합니다. "주문 취소해주세요"와 "결제 취소해주세요"처럼 구조가 비슷하지만 의미가 다른 문장을 384차원 공간에서 충분히 구분하지 못했습니다.

**해결**: MPNet-base-v2로 교체하여 768차원 벡터를 생성합니다. 2배 넓은 벡터 공간에서 더 세밀한 의미 차이를 포착할 수 있습니다.

**모델 비교**:
| 모델 | 차원 | 파라미터 | 특징 |
|------|------|----------|------|
| MiniLM-L12-v2 | 384 | 118M | 경량, 빠름, 기본 성능 |
| MPNet-base-v2 | 768 | 278M | 고성능, 느림, 세밀한 의미 구분 |

**이 모델을 선택한 이유**: sentence-transformers 라이브러리에서 다국어를 지원하면서 가장 높은 성능을 보이는 모델입니다. 한국어 데이터에서 MiniLM 대비 의미 구분 능력이 뛰어나며, 같은 파이프라인 구조에서 모델만 교체하면 되므로 코드 변경이 최소화됩니다.

#### 개선 2: "업무처리" 카테고리 제거

**문제 진단**: v2에서 업무처리의 정확도가 43.0%로 가장 낮았습니다. 혼동 행렬을 분석하면 업무처리가 결제(33건), 교환(29건), 반품(17건), 배송(17건)으로 **골고루 오분류**되었습니다. 이는 단순히 모델이 못 배운 것이 아니라, 카테고리 자체에 문제가 있음을 시사합니다.

**데이터 분석으로 확인**: 업무처리 카테고리의 `고객의도` 필드를 분석한 결과:

```
=== 업무처리 카테고리 (총 201,888건, 의도 16,051건) ===
  [647회] 상담사의도파악
  [589회] 방송상품주문        ← 주문과 동일
  [445회] 상품주문신청        ← 주문과 동일
  [365회] 주문요청            ← 주문과 동일
  [246회] 방송상품문의
  [220회] 상품가격문의
  [213회] 반품요청            ← 반품과 동일
  [197회] 고객불만제기
  [195회] 상품주문요청        ← 주문과 동일
  [177회] 방송상품주문요청    ← 주문과 동일
```

업무처리의 top 의도가 "방송상품주문", "상품주문신청", "주문요청" 등으로 **주문 카테고리의 의도와 거의 동일**했습니다. 즉, 업무처리는 독립적 의미를 가진 카테고리가 아니라 **다른 카테고리의 행정적 후속 절차**(접수, 처리, 확인)를 모아놓은 catch-all 카테고리였습니다.

**제거 결정의 근거**:

1. 업무처리의 고객 의도가 다른 카테고리(특히 주문)와 본질적으로 동일
2. AI가 의미적으로 구분할 수 없는 카테고리를 억지로 학습시키면 **다른 카테고리까지 성능 저하**
3. 카테고리 병합(업무처리→주문)도 고려했으나, 라벨링 기준 자체가 모호하므로 **제거**가 더 깔끔

**카테고리 병합을 택하지 않은 이유**:

- 업무처리 안에는 주문 외에도 반품, 교환 관련 의도가 섞여 있어 단순히 주문에 합칠 수 없음
- 각 건을 수작업으로 재라벨링하는 것은 비용 대비 효과가 낮음
- 제거하면 나머지 6개 카테고리의 경계가 자연스럽게 명확해짐

#### v3 결과: 카테고리별 성능 비교

| 카테고리 | v1 f1 | v2 f1 | **v3 f1** | v2→v3 변화 |
| -------- | ----- | ----- | --------- | ---------- |
| 교환     | -     | 0.77  | **0.83**  | +0.06      |
| 반품     | -     | 0.75  | **0.84**  | +0.09      |
| AS       | -     | 0.70  | **0.77**  | +0.07      |
| 결제     | -     | 0.66  | **0.74**  | +0.08      |
| 배송     | -     | 0.64  | **0.68**  | +0.04      |
| 주문     | -     | 0.44  | **0.55**  | +0.11      |
| 업무처리 | -     | 0.49  | (제거됨)  | -          |

**모든 카테고리에서 f1-score가 향상**되었습니다. 특히 주문은 +0.11로 가장 큰 폭의 개선을 보였습니다.

#### 테스트 예측 품질 변화

```
"주문한 물건이 아직 안 왔어요. 배송 조회 좀 해주세요."
  v2: 주문(0.309) ← 오분류
  v3: 배송(0.825) ← 정확 분류 ✅

"이 제품 환불하고 싶어요. 반품 접수해주세요."
  v2: 반품(0.840)
  v3: 반품(0.989) ← 확신도 대폭 상승 ✅

"카드 결제가 안 돼요. 다른 결제 수단으로 바꿀 수 있나요?"
  v2: 결제(0.893)
  v3: 결제(0.990) ← 확신도 대폭 상승 ✅
```

#### 이 방법들을 선택한 이유

v2에서 데이터 보강과 분류기 비교를 이미 시도했으나 +3.6%p에 그쳤습니다. 이는 **분류기 위(downstream)**가 아니라 **아래(upstream)**에 문제가 있음을 의미합니다:

1. **임베딩 벡터의 표현력 부족** → 더 큰 모델로 해결
2. **카테고리 정의 자체의 모호함** → 문제 카테고리 제거로 해결

이 두 가지는 분류기를 아무리 바꿔도 해결할 수 없는 근본적 문제였기에, 파이프라인의 상류(임베딩 모델, 카테고리 정의)를 개선하는 방향을 선택했습니다.

### 남아있는 한계 & 추가 개선 방향

현재 v3(73.9%)는 **사전학습 임베딩 + 전통 분류기** 파이프라인 내에서 거의 최선의 성능입니다. 주문 카테고리(47.5%)가 여전히 낮은 것은 "주문했는데 결제가 안 됐어요", "주문했는데 배송이 안 와요"처럼 **고객이 주문이라는 단어를 쓰면서 실제로는 결제/배송 문제를 말하는** 데이터 라벨링의 근본적 모호함 때문입니다.

더 높은 성능이 필요하다면 근본적으로 다른 접근법이 필요합니다:

| 방법                                             | 예상 효과 | 난이도 | 비용     |
| ------------------------------------------------ | --------- | ------ | -------- |
| Fine-tuning (임베딩 모델을 우리 데이터로 재학습) | 80~90%+   | 높음   | GPU 필요 |
| Cross-encoder (텍스트+카테고리를 함께 입력)      | 85%+      | 중간   | 느림     |
| LLM 기반 분류 (GPT/Claude에 직접 분류 요청)      | 90%+      | 낮음   | API 비용 |

이 프로젝트는 **교육용 실습**이므로, 임베딩과 분류기의 기본 원리를 이해하고 모델 개선 과정을 체험하는 것에 초점을 맞추었습니다. v1(60%) → v3(74%)까지의 개선 과정 자체가 학습의 핵심입니다.

## 데이터 전처리 파이프라인

원본 데이터는 콜센터 통화 녹취 텍스트이므로 다음과 같은 전처리를 적용합니다:

1. **대화셋 단위 결합** — 개별 문장이 아닌 `대화셋일련번호` 기준으로 고객 발화 + 상담원 응답(최대 3개) + 고객 의도를 하나로 합침
2. **ㅇㅇㅇ 익명화 토큰 제거** — 정규식 `ㅇ{2,}` 패턴 제거
3. **필러 표현 제거** — "네.", "아 그래요?", "여보세요?" 등 분류에 무관한 반응형 표현 9종 제거
4. **최소 길이 필터링** — 15자 미만 텍스트 제거 (분류 단서 부족)
5. **중복 제거** — 동일 텍스트 제거
6. **균등 샘플링** — 카테고리별 1,000건(학습), 200건(테스트)

## 모델 구조

```
고객 문의 텍스트
    ↓ clean_text() 전처리
    ↓ SentenceTransformer 임베딩 (768차원 벡터, MPNet)
    ↓ LogisticRegression vs SVC 비교 → 최선 선택
    ↓
예측 결과: 카테고리 + 확률 (Top-3)
```

## 예측 사용 예시

노트북의 모든 셀 실행 후, `predict_inquiry()` 함수로 문의 유형을 예측할 수 있습니다:

```python
predict_inquiry("주문한 물건이 아직 안 왔어요. 배송 조회 좀 해주세요.")
# → 배송 (0.825)

predict_inquiry("이 제품 환불하고 싶어요. 반품 접수해주세요.")
# → 반품 (0.989)

predict_inquiry("카드 결제가 안 돼요. 다른 결제 수단으로 바꿀 수 있나요?")
# → 결제 (0.990)
```

입력 텍스트에도 학습 시와 동일한 전처리(`clean_text()`)가 자동 적용됩니다.
