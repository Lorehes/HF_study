# 부서별 요청사항 라우팅 분류기

한국어 IT 헬프데스크 문의를 입력하면 **담당 부서(5개 카테고리)**를 자동으로 분류하는 AI 라우팅 시스템입니다.

## 프로젝트 구조

```
부서별_요청사항_라우팅/
├── korean_it_helpdesk.ipynb                          # 메인 노트북 (전처리 → 학습 → 예측)
├── README.md
└── Data/
    ├── make_data.py                                  # 학습 데이터 생성 스크립트 (v2)
    ├── korean_it_helpdesk_20000_with_5labels_v1.csv  # v1 데이터 (15개 템플릿)
    └── korean_it_helpdesk_20000_with_5labels_v2.csv  # v2 데이터 (300+ 조합형 패턴)
```

## 모델 평가

### 99.9% 정확도에 대한 해석 -- 템플릿 데이터의 한계

> **이 모델의 99.9% Accuracy는 실무 성능이 아닙니다.**

학습 데이터가 `make_data.py`로 **조합형 템플릿 기반 생성**되었기 때문에, 각 카테고리별 텍스트 패턴이 매우 규칙적입니다. 모델이 "IT 문의의 의미를 이해"한 것이 아니라, **템플릿의 반복 패턴을 학습**한 것입니다. 데이터를 직접 만들어서 학습시킨 것이므로 이 정도 수치는 예상된 결과입니다.

**실제 IT 헬프데스크 문의의 특성:**

- 오타, 구어체, 비표준 표현이 섞임 ("컴터가 맛갔어요", "프린터 안됨 ㅠ")
- 하나의 문의에 여러 카테고리가 걸치는 경우가 많음
- 학습 데이터에 없는 새로운 장비/소프트웨어가 지속 등장
- 같은 증상도 사람마다 완전히 다른 표현 사용

**실무 적용 시 예상 Accuracy: 70~85% 수준**으로 크게 하락할 것으로 예상됩니다.

### 강점

1. **압도적인 정량 성능** -- 5진 분류에서 99.9% Accuracy, 2,000건 중 단 2건만 오분류
2. **명확한 문의에 대한 확실한 분류** -- 카테고리 키워드가 뚜렷한 문의(키보드, 와이파이 등)에 100% 확률로 정확하게 분류
3. **SVM이 LR보다 확연히 우세** -- LR 98.9% vs SVM 99.9%로, 비선형 분류기의 장점이 드러남 (이커머스 감성 분석기에서는 LR이 우세했던 것과 정반대)
4. **복합 키워드 처리 가능** -- "백신에서 차단 + 컴퓨터도 느려짐" 같은 복합 문의에서도 주요 키워드를 기반으로 일관된 판단

### 한계점

1. **템플릿 데이터 의존** -- 99.9%는 템플릿 패턴 학습의 결과이며, 실제 사용자 문의에서는 큰 폭으로 하락 예상. 데이터를 직접 생성했으므로 높은 수치는 예상된 것
2. **학습 데이터에 없는 표현에 대한 확신도 편차** -- 학습 패턴과 유사한 "키보드 눌리지 않아요"는 100%이지만, 학습 패턴에 없는 "노트북 충전이 안 됩니다"는 69.2%로 확신도가 떨어짐
3. **Inquiry/Help 카테고리의 과대 흡수** -- "ERP에서 DB 조회가 안 됨"이 Database(13.6%)가 아닌 Inquiry/Help(81.5%)로 분류됨. "~인가요?" 같은 질문형 패턴이 Inquiry에 학습된 결과
4. **카테고리 간 데이터 불균형** -- Hardware(5,082건) vs Database(2,420건)로 2배 이상 차이. Database의 표현 다양성이 부족하여 새로운 DB 문의에 취약

### 실전 테스트 결과

```
"컴퓨터가 부팅이 안 되고 팬 소리만 크게 나요. CPU 과열 같은데 확인 좀 해주세요."
  → Hardware ✅

"윈도우 업데이트 후 프로그램이 실행이 안 돼요. 호환성 문제인 것 같습니다."
  → Software ✅

"VPN 연결이 자꾸 끊겨서 재택근무가 안 됩니다. 오류 코드 800이 떠요."
  → Network ✅

"Oracle DB 쿼리 실행 속도가 갑자기 느려졌어요. 인덱스 점검 부탁드립니다."
  → Database ✅

"백신 프로그램 라이선스가 만료됐는데 갱신하려면 어떻게 해야 하나요?"
  → Inquiry/Help ✅
```

**경계가 모호한 문의:**

```
"공유기를 새로 교체했는데 랜선을 꽂아도 인터넷이 안 잡혀요. 랜 포트 불량인가요?"
  → Network (1.000) -- HW 가능성 있지만 "인터넷", "공유기" 키워드가 지배적

"사내 ERP 프로그램에서 데이터 조회가 안 되는데 DB 서버가 다운된 건가요?"
  → Inquiry/Help (0.815) -- DB(0.136)보다 질문형 패턴이 강하게 작용

"컴퓨터가 느려서 프로그램도 안 열리고 인터넷도 끊기는데 원인을 모르겠어요."
  → Hardware (0.479) vs Network (0.430) -- 4.9%p 차이, 분류기가 실제로 "헷갈림"
```

### 추가 개선 방향

| 방법                                           | 예상 효과                                       | 비고             |
| ---------------------------------------------- | ----------------------------------------------- | ---------------- |
| **실제 IT 헬프데스크 로그로 학습 데이터 교체** | 실무 수준의 정확도 측정 가능                    | 가장 중요한 개선 |
| 서브카테고리(15개) 분류로 확장                 | "Hardware > CPU" vs "Hardware > Monitor" 세분화 | 난이도 상승      |

## 데이터 전처리 파이프라인

IT 헬프데스크 문의 데이터는 템플릿 기반 생성이므로 이커머스 리뷰보다 깨끗합니다:

1. **공백 정리** -- 다중 공백 → 단일 공백, 앞뒤 공백 제거
2. **최소 길이 필터링** -- 10자 미만 텍스트 제거 (분류 단서 부족)
3. **중복 제거** -- 동일 텍스트 제거 (과적합 방지)
4. **균등 샘플링** -- 카테고리당 1,500건(학습용), 400건(평가용)

**전처리 결과:**

- 원본: 20,000건 → 전처리 후: 거의 변동 없음 (템플릿 데이터이므로 노이즈 최소)
- 학습 데이터: 7,500건 (카테고리당 1,500 x 5)
- 평가 데이터: 2,000건 (카테고리당 400 x 5)

## 모델 구조

```
IT 문의 텍스트
    ↓ clean_text() 전처리
    ↓ SentenceTransformer 임베딩 (768차원 벡터, MPNet)
    ↓ LogisticRegression vs SVC 비교 → 최선 선택
    ↓
예측 결과: 5개 카테고리 확률 분포 → 최고 확률 카테고리 선택
```


| 데이터 특성     | 실제 통화 녹취               | 실제 사용자 리뷰               | **인위적 생성 데이터** |
| 전처리          | ㅇㅇㅇ 제거, 필러 제거, 15자 | HTML 제거, 반복 문자 축소, 5자 | 공백 정리, 10자        |

| 최종 Accuracy   | 73.9%                        | 81.8%                          | 99.9%                  |
| **성능 신뢰도** | **높음** (실제 데이터)       | **높음** (실제 데이터)         | **낮음** (생성 데이터) |

> 이커머스 감성 분석기의 81.8%나 고객문의 분류기의 73.9%가 더 "진짜 성능"입니다.
> 실제 사용자가 작성한 데이터를 사용했기 때문에 실무에서도 유사하게 재현될 가능성이 높습니다.
> 반면 IT 헬프데스크의 99.9%는 템플릿 데이터의 규칙성이 반영된 수치이므로, 실무 적용 시 큰 폭으로 하락합니다.
